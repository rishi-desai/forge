Verdict at a glance

The offline math layer is solid: all 104 tests pass (51 core + 30 ML + 23 regime), the sizing formula matches the spec's worked examples, the settled-cash T+1 ledger is correct, the universal $10k cap genuinely binds everything, and the ML/regime fallback design is exactly as documented — every failure path degrades to rules.

The problems are concentrated in the live wiring between layers — the orchestrator/app seam that no test covers. Three of the four critical findings mean the bot, as deployed, will not behave the way the README promises: approved high-conviction trades are silently dropped, positions can effectively never exit, and the public Cloudflare tunnel exposes unauthenticated control endpoints. None of these are visible in tests because the tests deliberately stop at the pure-math boundary.

1. Approved (or expired) conviction overrides never execute a trade

backend/orchestrator.py:255-265, 302-309 · backend/app.py:241-260 · frontend/src/App.jsx:78-82

When an ALL-IN (or any approval-gated tier) fires, execute_candidate() emits the override_pending event and returns, discarding the sized candidate. The comment says "executed on approval (app.py) or as normal size on expiry" — but neither happens:

  • app.py approve_override() calls resolve_pending_override(), which just pops the override and returns it. No code path ever calls _finalize_and_submit() with the approved override. The TradeCandidate isn't even retained — _pending_payloads holds only the display dict.
  • On expiry, run_forever() logs "reverting to normal sizing" but never submits the normal-size trade either.
  • The dashboard makes it worse: on approve, the UI toasts "Executing at override size" — the user believes a 50%-of-cash trade went in. It didn't. The override is also never recorded via record_execution(), so rate-limit counters stay wrong too.

This is the exact opposite of the spec's intent: the system's highest-conviction setups are the only ones that never trade.

Fix options
Option A (recommended): retain the full execution context server-side. Store {candidate, normal_size, open_positions_snapshot} keyed by approval_id on the Orchestrator (not in app.py). On approve, call _finalize_and_submit(cand, override_size, …) from a method the API invokes (guard with a lock since it runs on the API thread while the bot loop runs). On expiry in expire_stale(), submit at normal_size as documented. Re-validate freshness first — if the quote moved or >N minutes passed, re-run check_capacity and re-price.
Option B (simpler, behavior change): drop deferred execution entirely. On approval-required tiers, trade immediately at normal size, and treat approval as an upsize of the open position (buy more contracts). No stored state, no staleness problem — but two orders instead of one.
Either way: add an orchestrator-level test that walks signal → pending → approve → order submitted (with a dry-run broker), and one for expiry → normal-size order. This whole seam is currently untested.

2. Position exits are effectively dead — nothing ever closes except 0DTE

backend/orchestrator.py:313-361 (manage_positions, _estimate_mark)

_estimate_mark() returns cost unconditionally (drift = 0.0), so pnl is always 0 for every open position. Consequences, chained:

  • Profile stop-losses never fire (pnl <= -stop × cost is never true).
  • The 50%-of-credit profit target never fires.
  • The 8% equity stop never fires.
  • There is no expiry handling at all — an option past its expiry date stays status='open' in the DB forever. Deployed capital never frees, so the concurrency, correlation, and deployment caps progressively choke off all new trading. On the Alpaca side the contracts expire/exercise, so the local ledger and the broker permanently diverge.

The README discloses the conservative mark ("positions held at cost") but not the implication that every management rule in section 7 is inert. The learning loop (Layer 5) also starves: with no closed trades there are no lessons, no weight updates, and the ML engine never reaches its 30-trade bootstrap.

Fix options
Option A (recommended): wire real option marks. You already depend on alpaca-py; use OptionHistoricalDataClient.get_option_latest_quote() with the OCC symbols stored in legs_json to compute a true per-position mark (sum legs, signed by side). This also fixes the conviction criteria and unrealized P&L on the dashboard.
Option B (interim, zero new APIs): approximate the mark with the Black-Scholes machinery you already have — reprice each leg with bs_price() using the current underlying quote, remaining DTE, and the entry IV proxy. Imperfect, but it makes stops/targets/learning live today instead of never.
Required regardless: add an expiry sweep to manage_positions() — any trade with expiry < today gets closed at intrinsic (or 0) and a db.close_trade(). Also reconcile against broker.positions() daily so local state can't drift from Alpaca.

3. Public Cloudflare tunnel exposes unauthenticated control endpoints

cloudflared.yml · backend/app.py:38-43, 218-236, 241-260 · Makefile (tunnel targets)

The tunnel publishes api.forge.rishid.com → localhost:8000. The FastAPI app has no authentication of any kind. Anyone who finds the hostname can:

  • PUT /api/settings — switch your risk profile to max_aggression, change starting cash, rewrite custom_overrides and conviction_overrides (persisted to config.json on disk).
  • POST /api/overrides/{id}/approve — approve an ALL-IN (50% of cash) trade meant for you.
  • POST /api/ml/retrain, POST /api/reports/weekly/run — burn CPU at will.
  • Read every trade, signal, weight, and event via the GET endpoints and the open WebSocket.

CORS is not a defense here — it only constrains browsers, not curl. Today it's paper money and a config file; it's still your API keys' rate budgets, your disk, and a system you intend to trust with real-money decisions someday.

Fix options
Option A (recommended, zero code): put Cloudflare Access in front of the hostname (free for personal use) — one-click SSO/email-pin gate at the edge; the backend stays untouched.
Option B (app-level): add a bearer-token check — API_TOKEN in .env, a FastAPI dependency that rejects mutating routes (and ideally all routes) without Authorization: Bearer …, and the matching header in frontend/src/api.js. ~20 lines.
Option C (minimum): if the tunnel is only for your own remote access, scope it: keep GETs public-ish but require the token on every PUT/POST, and rename the hostname to something unguessable. Weaker than A/B.

4. close_position() targets the underlying symbol, not the option legs

backend/orchestrator.py:344 · backend/alpaca_exec.py:125-133

manage_positions() calls self.broker.close_position(t["symbol"], …) with the underlying ticker (e.g. "SPY"). In non-dry-run mode that becomes TradingClient.close_position("SPY") — but the actual Alpaca positions are the OCC option contracts (e.g. SPY260619P00585000 × 4 legs for a condor). The call will 404 (no SPY equity position) or, worse, liquidate an unrelated SPY share position. Meanwhile the local DB marks the trade closed and credits the ledger with estimated proceeds — full divergence from the broker.

Fix options
Option A (recommended): close by legs. Parse legs_json from the trade row and submit the inverse order: single-leg → close_position(occ_symbol); multi-leg → an MLEG order with PositionIntent.SELL_TO_CLOSE/BUY_TO_CLOSE mirroring the open order (you already build MLEG opens in submit_candidate, so this is symmetric code).
Also: only call db.close_trade() and ledger.receive_proceeds() after the close order is accepted (check the returned status), and use the actual fill value as proceeds when available rather than the estimate.

5. Weekly-loss size-halving can never trigger

backend/orchestrator.py:379-381 · backend/risk.py:352-356

run_forever() calls self.guards.start_day(eq) every day with no week_start_equity, so GuardState.week_start_equity is reset to today's equity each morning. The Wednesday weekly-drawdown check then compares today against today — always ~0%. The README's promised "Wednesday weekly-loss size-halving" is dead in live operation (the unit test passes because it injects week_start_equity manually).

Fix
Fix: persist Monday's opening equity. On day-roll, if now.weekday() == 0 (or no stored value for the current ISO week), write the equity to the DB (e.g. an events/kv row or reuse equity_curve's first Monday snapshot) and pass it: self.guards.start_day(eq, week_start_equity=monday_eq). Persisting also survives restarts, which the current in-memory GuardState doesn't.

6. Equity calculation ignores unsettled proceeds → false daily-drawdown halts

backend/orchestrator.py:197-199, 379-380 · backend/risk.py:241-275

Equity is computed as ledger.available() + deployed. When a position closes, its cost leaves deployed immediately but the proceeds sit in ledger.unsettled until T+1 — invisible to this formula. Closing a $1,000 position on a $10,000 account makes measured equity drop 10% instantly, which trips the daily-drawdown circuit breaker on the aggressive profile and halts trading for the day even on a profitable close. The same distortion feeds snapshot_equity(), so the dashboard equity curve dips on every close and recovers the next day.

Fix
Fix: expose pending cash on the ledger — def total(self): return self.settled + sum(amt for _, amt in self.unsettled) — and use ledger.total() + deployed for equity/guards/snapshots, keeping available() strictly for spend-validation. One method, three call sites.

7. Override sizing double-applies the weekly size-halving multiplier

backend/orchestrator.py:248-251, 267-269 · backend/risk.py:216-229

normal is computed with guards.size_multiplier() applied, and that value is stored as ov.normal_size. For STRONG/MAX, compute_override_size() derives the override size from ov.normal_size — already halved — and then line 269 multiplies by the guard multiplier again. When sizes are halved (once #5 is fixed), overrides would trade at 25% instead of 50%. Latent today only because #5 means the multiplier is never 0.5.

Fix
Fix: apply the multiplier exactly once at the end. Pass the unhalved normal size into overrides.evaluate() (it's also what the spec's tier math expects as "normal size"), and multiply the final sized by size_multiplier() in one place. Add a test: halved guards + STRONG override ⇒ exactly normal × 1.5 × 0.5.

9. Rejected-override log inserts duplicates on every evaluation

backend/orchestrator.py:252-254 · backend/overrides.py:216-223

execute_candidate() runs for rec in self.overrides.rejected_log[-3:]: db.log_rejected_override(rec) after every evaluation, re-inserting the last three in-memory rejections each time any candidate is evaluated. After a day of scans, one real rejection becomes hundreds of DB rows, and the dashboard's "rejected overrides" view becomes noise. rejected_log also grows unbounded in memory.

Fix
Fix: make it a drain: while self.overrides.rejected_log: db.log_rejected_override(self.overrides.rejected_log.pop(0)) — or better, have _log_rejection() write to the DB directly and delete the in-memory list entirely (db is already imported everywhere else).

10. The 0.30 composite threshold is vacuous on the rules path

backend/signals.py:320-321 · backend/orchestrator.py:221 · backend/risk.py:139

strength = 0.30 + 0.70 × (…) — the formula's floor equals the trade threshold, so any single fired rule (including a lone 0.45-strength contrarian macro signal) clears both the comp["strength"] < 0.30 gate and MIN_SIGNAL_STRENGTH_TO_TRADE. The README's "Composite < 0.30 → no trade" filter never filters anything. Note the 0dte_vwap_trend rule fires on SPY/QQQ on essentially every scan after 10:30 (price is always above or below the SMA20 proxy), so combined with this floor the bot will steadily open 0DTE spreads until the concurrency cap absorbs them. Only the ML path (raw probability, can be < 0.30) makes the gate meaningful.

Fix options
Option A: remove the additive floor — strength = 0.70 × (0.5 × dominance + 0.5 × magnitude) recalibrated so multi-signal confluence lands where 0.30 means something — then re-derive the spec worked-example tests.
Option B (smaller blast radius): keep the formula, raise the trade gate to a calibrated value (e.g. 0.45) and/or require ≥2 fired signals from ≥2 categories before select_strategy runs. Also give 0dte_vwap_trend a real trend condition (distance from VWAP, persistence) so it isn't a coin-flip generator.

11. A pending ALL-IN re-queues a brand-new approval modal every scan

backend/overrides.py:141-191 · backend/orchestrator.py:207-245

While an approval is pending, the next 5-minute scan re-evaluates the same symbol, qualifies again, and evaluate() happily creates a second pending override with a new ID (rate limits only count executed overrides). The user gets stacked modals for the same trade; approving an old one (post-#1 fix) could double-execute.

Fix
Fix: in evaluate(), return None (or the existing pending override) when self.pending already contains an entry for the same symbol+tier. One containment check.

13. Makefile starts the live bot under --reload, and RUN_BOT is hidden coupling

Makefile:74-82 · .env (RUN_BOT=1) · backend/app.py:84-97

make start-backend runs uvicorn app:app --port 8000 --reload. Because RUN_BOT=1 lives in .env, this is the live bot — and any .py edit restarts the process mid-session, wiping all in-memory state: daily/weekly override counters, drawdown guard state, pending approvals, _open_meta hold timers. A halted day silently un-halts after an edit. Nothing in make start's output tells you whether you launched a dashboard or a trading bot.

Fix
Fix: drop --reload from the start target (add a separate make dev with reload and RUN_BOT=0), set RUN_BOT explicitly per target (RUN_BOT=1 $(UVICORN) … for start), and print which mode launched. Longer term, persist guard/override counters to the DB so restarts are harmless (also fixes #5's restart amnesia).

14. Finnhub candle fallback retries the dead endpoint on every call

backend/data_clients.py:101-111

Finnhub's free tier 403s /stock/candle, so every candles() call burns a token-bucket slot and an HTTP round-trip on a guaranteed failure before falling back to Alpaca — and the Alpaca result is not cached (the TTL cache only wraps the Finnhub attempt). Per scan: 13 symbols × every 5 minutes, plus the 560-day SPY regime fetch — all uncached Alpaca calls and wasted Finnhub budget that the quote/news calls actually need.

Fix
Fix: cache at the function boundary instead of inside the Finnhub branch: cached(f"c:{symbol}:{resolution}", 300, fetch_finnhub_then_alpaca). Add negative-result memory for the Finnhub endpoint (e.g. after the first 403, skip straight to Alpaca for 24h via a module flag or cache key).

15. Settings PUT can persist config that crashes the next bot boot

backend/app.py:218-236 · backend/risk.py:152-161 · backend/orchestrator.py:50-51

put_settings writes custom_overrides to config.json unvalidated. The validation (get_profile raising on unknown keys) only runs at Orchestrator construction — i.e., the next restart, which then crash-loops on startup with the bad config persisted. 16. Weekly Sharpe annualizes 5-minute equity snapshots as daily returns

backend/learning.py:101-103, 63-70 · backend/orchestrator.py:403-405

equity_curve rows are written every ~5-minute scan, but generate_weekly_report() treats consecutive snapshots as "daily_returns" and annualizes with √252. The reported Sharpe is wrong by roughly √78 (≈8.8×) in scale, and max-drawdown is intraday rather than the daily figure the report implies.

Fix: resample to one equity point per day before computing returns — group curve by ts[:10] and take the last snapshot of each day. Three lines in generate_weekly_report().

backend/options_math.py:128-133 · backend/strategies.py:95-106
